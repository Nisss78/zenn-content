# OWASP×Microsoft：AIエージェントのガバナンスがついに始まる

2026年4月2日、Microsoftが**Agent Governance Toolkit（AGT）**をオープンソースで公開した。これはAIエージェントの実行時セキュリティを担保する初の包括的ツールキットであり、2025年12月にOWASPが発表した「**Agentic AI Top 10**」の全リスクに対応している。

AIエージェントはもはやチャットで質問に答えるだけの存在ではない。フライトの予約、コードの実行、インフラの管理、取引の遂行――自律的に行動するエージェントが当たり前になる中で、「**誰がエージェントの振る舞いを監視するのか**」という問いがついに具体化した。

本記事では、このAgent Governance Toolkitの設計思想、OWASP Agentic AI Top 10との対応、そして開発者が今日から使える実践的な導入方法を解説する。

---

## なぜ今、エージェントのガバナンスが必要なのか

### エージェントの自律性がもたらすリスク

LangChain、AutoGen、CrewAI、OpenClawといったフレームワークのおかげで、自律型AIエージェントを構築することはかつてないほど簡単になった。しかし、容易に構築できるということは、**容易に危険なエージェントを構築できる**ことも意味する。

具体的なリスクシナリオを考えてみよう：

- **目標ハイジャック**：プロンプトインジェクションにより、エージェントの目的が乗っ取られる
- **ツール悪用**：エージェントが与えられた権限以上の操作を実行する
- **記憶汚染**：エージェントの長期記憶に悪意ある情報が混入する
- **カスケード障害**：複数エージェント間でエラーが連鎖する
- **ローグエージェント**：制御を離れたエージェントが予期せぬ行動をとる

これらは理論上の脅威ではない。既に実際の運用環境で報告されている問題だ。

### 規制の波も迫っている

- **EU AI Act**：2026年8月から高リスクAIシステムの義務が施行
- **Colorado AI Act**：2026年6月から執行開始
- **OWASP Agentic AI Top 10**：2025年12月に初の正式なリスク分類を発表

インフラが規制に追いついていない。Microsoftはこの隙間を埋めるためにAGTを構築した。

## Agent Governance Toolkitの全体像

### 設計思想：OSカーネルからの学び

AGTの最も興味深い設計思想は、**オペレーティングシステムのカーネル設計**をAIエージェントに適用した点だ。

- OSは「複数の信頼できないプログラムがリソースを共有する環境」を管理するために、特権リング、プロセス分離、システムコールインターセプトを発明した
- サービスメッシュは、マイクロサービス間の通信を保護するためにmTLSとアイデンティティ管理を導入した
- SREは分散システムの信頼性を担保するためにSLOとサーキットブレーカーを確立した

AGTは、これらの「すでに解決済みの問題」に対する答えをAIエージェントに持ち込んだ。

### 7つのパッケージ構成

AGTはモノレポ構造で、7つの独立してインストール可能なパッケージから成る：

1. **Agent OS**：ステートレスなポリシーエンジン。全エージェントのアクションを実行前にインターセプト（サブミリ秒レイテンシ）
2. **Identity**：DIDベースのアイデンティティ管理と行動トラストスコアリング
3. **SRE**：サーキットブレーカー、SLO管理、自動フェイルオーバー
4. **MCP Security Gateway**：Model Context Protocolのセキュリティゲートウェイ
5. **Cross-Model Verification Kernel（CMVK）**：複数モデルによる多数決検証
6. **Inter-Agent Trust Protocol（IATP）**：エージェント間通信の暗号化レイヤー
7. **Plugin System**：Ed25519署名付きプラグインとマニフェスト検証

Python、TypeScript、Rust、Go、.NETの5言語に対応している。

## OWASP Agentic AI Top 10との対応

OWASPが定義した10のリスクカテゴリに対して、AGTがどう対応するかを整理する：

| OWASPリスク | AGTの対応メカニズム |
|---|---|
| 目標ハイジャック | ポリシーエンジンの意味的インテント分類 |
| ツール悪用 | ケイパビリティサンドボックス + MCPセキュリティゲートウェイ |
| アイデンティティ悪用 | DIDベースアイデンティティ + 行動トラストスコア |
| サプライチェーンリスク | Ed25519署名付きプラグイン + マニフェスト検証 |
| コード実行の危険性 | 実行リング + リソース制限 |
| 記憶汚染 | CMVK（複数モデル多数決検証） |
| 通信の脆弱性 | IATP暗号化レイヤー |
| カスケード障害 | サーキットブレーカー + SLO管理 |
| 人間-エージェント信頼悪用 | クオラムロジック付き承認ワークフロー |
| ローグエージェント | リング分離 + トラスト減衰 + 自動キルスイッチ |

### 特に注目すべき3つのメカニズム

#### 1. ポリシーエンジン（Agent OS）

エージェントがアクションを実行する直前に、全てのツールコールをインターセプトし、ポリシーに照らして許可・拒否を判断する。ステートレス設計により、パフォーマンスへの影響はサブミリ秒。セキュリティを「オプションのラッパー」ではなく「実行パスの一部」として組み込むアプローチだ。

#### 2. Cross-Model Verification Kernel（CMVK）

エージェントの記憶に書き込まれる情報を、複数の異なるLLMで検証する仕組み。あるモデルの出力を別のモデルが検証する「多数決」アプローチにより、単一モデルのハルシネーションや悪意ある出力を検出できる。

#### 3. Inter-Agent Trust Protocol（IATP）

マルチエージェント環境で、エージェント間の通信を暗号化し、信頼スコアに基づいてアクセスを制御する。信頼スコアは時間とともに減衰する設計で、長期間監視されていないエージェントの権限が自動的に縮小される。

## 実践：5分で導入するエージェントガバナンス

### インストール

```bash
pip install agent-governance-toolkit[full]
```

フレームワーク別の個別インストールも可能：

```bash
# LangChain用
pip install agent-governance-toolkit[langchain]

# CrewAI用
pip install agent-governance-toolkit[crewai]

# OpenAI Agents SDK用
pip install openai-agents-governance
```

### LangChainでの基本設定例

```python
from agent_os import PolicyEngine, ToolCallInterceptor
from agent_os.integrations.langchain import GovernanceCallback

# ポリシーエンジンの初期化
engine = PolicyEngine(
    policies=[
        "block_file_deletion",
        "limit_api_calls_per_minute:100",
        "require_approval_for:database_write"
    ]
)

# LangChainエージェントにガバナンスを適用
governance = GovernanceCallback(engine)

agent = Agent(
    tools=[file_tool, api_tool, db_tool],
    callbacks=[governance]  # これだけ
)
```

たったこれだけで、エージェントの全アクションがポリシーエンジンを通過するようになる。

### ポリシーのカスタマイズ

```python
from agent_os import Policy, Severity

# カスタムポリシーの定義
custom_policies = [
    Policy(
        name="sensitive_data_protection",
        description="個人情報を含む可能性のあるAPIレスポンスのマスキング",
        severity=Severity.CRITICAL,
        action="mask_pii"
    ),
    Policy(
        name="cost_control",
        description="1セッションあたりのAPI呼び出し上限",
        severity=Severity.WARNING,
        action="throttle",
        threshold=500
    )
]

engine = PolicyEngine(policies=custom_policies)
```

## フレームワークエコシステムとの統合状況

AGTは主要なエージェントフレームワークとの統合を既に提供している：

- **Dify**：ガバナンスプラグインがマーケットプレイスに公開済み
- **LlamaIndex**：TrustedAgentWorkerとして統合済み
- **OpenAI Agents SDK**：PyPIで公開済み
- **LangGraph**：PyPIで公開済み
- **Haystack**：アップストリームに統合
- **PydanticAI**：ワーキングアダプター提供
- **CrewAI**：タスクデコレータ経由で統合

各統合はフレームワークのネイティブな拡張ポイント（LangChainのコールバックハンドラ、CrewAIのタスクデコレータなど）を使うため、エージェントコードの書き直しは不要だ。

### OpenClawでの活用可能性

OpenClawのスキルシステムとMCP統合は、Agent OSのポリシーエンジンと非常に相性が良い。OpenClawのツール呼び出しをPolicyEngineでインターセプトすることで、スキルの実行前にセキュリティチェックを挟むことができる。特に、SlackやDiscordなどのメッセージングチャネル経由で動くエージェントでは、送信前のコンテンツフィルタリングが重要だ。

## プロトコルレイヤーの動向：ACP、A2A、MCP

AGTの登場と並行して、エージェント間通信のプロトコル層も急速に整理されている：

- **ACP（Agent Communication Protocol）→ A2A（Agent-to-Agent）**：Linux Foundation下で統合。Google ADKがネイティブ対応
- **MCP（Model Context Protocol）**：200+のサーバー実装を突破。Anthropicが開始し、OpenAIやGoogleも採用
- **2026 MCPロードマップ**：トランスポートのスケーラビリティ、エージェント間通信、ガバナンスの成熟化、エンタープライズ対応が重点課題

MCPは2024年11月のAnthropicの静かなオープンソース化から14ヶ月で、AI業界のエージェント・ツール間統合のバックボーンとなった。2,300以上のMCPサーバーが公開ディレクトリに登録されている。ただし、セキュリティ上の課題（ダブルホップレイテンシ、コンテキストウィンドウの肥大化）も指摘されており、AGTのようなガバナンス層の必要性が高まっている。

## 開発者が今日からできる5つのこと

1. **OWASP Agentic AI Top 10を読む**：https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/
2. **AGTをインストールしてポリシーエンジンを試す**：`pip install agent-governance-toolkit[full]`
3. **既存エージェントにコールバックを追加する**：フレームワークの拡張ポイントに数行追加するだけ
4. **サーキットブレーカーを有効にする**：カスケード障害の防止に最も効果的
5. **エージェントの権限を最小限にする**：実行リングとケイパビリティサンドボックスで制限

## まとめ：ガバナンスは「後付け」ではない

AIエージェントのガバナンスは、これまでセキュリティ業界の「後で考える」領域だった。しかし、エージェントが自律的に行動する世界では、ガバナンスはアーキテクチャの第一天井から組み込むべきものだ。

MicrosoftのAgent Governance Toolkitは、その最初の具体的な答えを出した。OSカーネル、サービスメッシュ、SREという数十年の知見をAIエージェントに適用したアプローチは、実用的で説得力がある。

エージェントを構築する全ての開発者にとって、今はガバナンスを「機能」ではなく「基盤」として捉え直すタイミングだ。
